# OM World Governance

## Status

This document is a public draft.

OM World is currently in the Genesis phase.
The governance model described here is not final. It is a starting framework for discussion, contribution, iteration, and progressive decentralization.

## One Mind, One World.

OM World is a decentralized intent economy protocol.

Its governance must protect the core principle of the protocol:

**Humanity owns its intentions.**

Governance exists to preserve the open, verifiable, and user-sovereign nature of OM World.

It should not become a mechanism for capture, speculation, manipulation, or centralized control.

## Governance Purpose

The purpose of OM World governance is to coordinate the evolution of the protocol.

Governance should help decide:

* protocol standards
* intent schema updates
* agent mandate rules
* execution proof requirements
* tool registry rules
* verifier requirements
* settlement parameters
* dispute processes
* grant allocation
* contributor recognition
* security responses
* ecosystem priorities
* protocol upgrades

Governance should not control individual user intentions.

A user’s intent belongs to the user.

The protocol may define rules for how intentions are expressed, verified, and settled, but it should not become the owner of those intentions.

## Core Governance Principles

### 1. Intent Sovereignty

The originator owns the intention.

Governance must not create systems where user intentions can be captured, sold, manipulated, or redirected without explicit consent.

### 2. Verifiability Before Trust

Important protocol actions should be recorded, auditable, and explainable.

Governance decisions should produce public records.

Where possible, outcomes should be backed by:

* signed proposals
* vote records
* execution logs
* implementation commits
* audit reports
* on-chain transactions
* public meeting notes

### 3. Open Protocol, Not Closed Platform

OM World should evolve as public infrastructure.

Governance must avoid concentrating power in a single company, foundation, model provider, interface, or privileged operator.

### 4. Progressive Decentralization

OM World should not pretend to be fully decentralized on day one.

The protocol should decentralize progressively as:

* contributors increase
* specifications mature
* security improves
* tools are adopted
* verifiers emerge
* governance mechanisms are tested

### 5. Security Before Scale

The governance system should prioritize protocol safety over fast growth.

No upgrade, incentive, or expansion should compromise:

* user authorization
* verification integrity
* funds in escrow
* tool registry trust
* dispute fairness
* protocol credibility

### 6. Anti-Capture

Governance must be designed to resist capture by:

* large token holders
* centralized platforms
* model providers
* tool monopolies
* verifier cartels
* capital providers
* founders
* short-term speculators
* coordinated attackers

### 7. Contributor Legitimacy

Governance should recognize meaningful contribution, not only capital ownership.

Useful work should matter.

Examples:

* protocol research
* code
* smart contract development
* security review
* tool creation
* verification design
* governance design
* documentation
* translation
* community building
* design
* ecosystem development

### 8. Transparency With Privacy

Governance should be transparent at the protocol level.

However, OM World must also respect user privacy.

Governance should not require exposing sensitive user data or private intentions unless the user explicitly consents or a dispute process requires controlled disclosure.

### 9. Slow on Values, Fast on Execution

Core values should change slowly.

Implementation details can evolve quickly.

OM World should be careful with foundational principles and pragmatic with technical iteration.

## Governance Non-Scope

Governance should not:

* own user intentions
* force users to reveal private intents
* secretly route intents to preferred tools
* sell or auction user intent
* override signed user mandates without due process
* manipulate agent decisions for hidden commercial benefit
* grant permanent privilege to any single entity
* promise financial returns
* use governance as a marketing instrument
* treat token price as the protocol’s primary objective

## Phase 3 — Reserved

**No token is planned in any phase of OM World's Genesis governance.** No allocation of any token or financial instrument is contemplated for any contributor or class of contributors. If, after extensive future review, a settlement asset is ever proposed, its governance model must be designed and reviewed through a separate, public process — independently of any contributor's status today and independently of any commitment made in this document.

Detailed token-governance design is out of scope for the Genesis phase. The relevant open research questions (voting models, staking, anti-capture mitigations) are noted in [LITEPAPER §9 Settlement Asset](LITEPAPER.md) and [discussions/genesis-questions.md](discussions/genesis-questions.md) — they are questions, not roadmap.

## Phase 4 — DAO Governance

### Objective

Move protocol control to decentralized governance.

### DAO Responsibilities

A mature OM World DAO may govern:

* protocol parameters
* treasury allocation
* grants
* registries
* verifier rules
* disputes
* upgrades
* security processes
* public goods funding
* ecosystem development

### DAO Limits

Even a DAO should not have unlimited power.

The DAO should not be able to:

* seize user intents
* alter private user records
* redirect user payments outside mandate rules
* secretly privilege specific tools
* remove verification requirements for insiders
* bypass signed user consent
* change immutable records retroactively

### Constitutional Constraints

OM World may eventually need a protocol constitution.

This constitution should define:

* non-negotiable principles
* user rights
* governance limits
* emergency limits
* upgrade rules
* dispute rights
* privacy protections
* anti-capture protections

## Proposal Process

OM World should use a clear proposal process.

## Proposal Types

### OIP — OM Improvement Proposal

For protocol-level changes.

Examples:

* update Intent Schema
* update Tool Registry standard
* add verifier requirements
* change settlement rules
* define dispute model

### OGP — OM Governance Proposal

For governance-level changes.

Examples:

* create working group
* change proposal process
* define voting model
* allocate treasury funds
* modify contributor recognition

### OSP — OM Security Proposal

For security-sensitive changes.

Examples:

* patch vulnerability
* pause contract
* change verifier slashing
* update emergency process
* respond to exploit

### ORP — OM Research Proposal

For research and exploration.

Examples:

* compare proof systems
* study agent abuse
* analyze decentralized compute
* map legal risks
* explore privacy models

## Proposal Template

Every proposal should include:

```markdown
# Proposal Title

## Proposal Type

OIP / OGP / OSP / ORP

## Status

Draft / Review / Accepted / Rejected / Implemented

## Summary

A short explanation of the proposal.

## Motivation

Why this proposal is needed.

## Specification

The proposed change in detail.

## Rationale

Why this approach is preferred.

## Alternatives Considered

Other options and why they were not chosen.

## Risks

Technical, governance, legal, economic, or social risks.

## Security Considerations

How this affects user safety, protocol integrity, or funds.

## Privacy Considerations

How this affects user data or intent privacy.

## Implementation Plan

Steps required to implement.

## Review Period

Expected review timeline.

## References

Links to related issues, pull requests, discussions, papers, or prior decisions.
```

## Proposal Lifecycle

### 1. Draft

A contributor submits an idea as an issue or discussion.

### 2. Review

The community reviews the proposal.

Working groups may provide comments.

### 3. Revision

The author updates the proposal based on feedback.

### 4. Decision

The relevant governance process accepts, rejects, or requests more work.

### 5. Implementation

Accepted proposals are implemented through pull requests, contracts, processes, or public documentation.

### 6. Record

The final decision and implementation record are preserved publicly.

## Decision-Making During Genesis

During the Genesis phase, decisions should follow this process:

1. Open a GitHub issue.
2. Allow public comments.
3. Draft the change.
4. Submit a pull request.
5. Request review from relevant contributors.
6. Merge when there is rough consensus.
7. Record important decisions in a decision log.

Genesis decisions do not require token voting.

The goal is clarity, speed, and public accountability.

## Rough Consensus

OM World may use rough consensus during early stages.

Rough consensus does not mean unanimous agreement.

It means:

* objections are heard
* tradeoffs are documented
* no major unresolved safety concerns remain
* affected contributors have had a chance to respond
* the decision is aligned with core principles

## Decision Log

Important decisions should be recorded in a future file:

```text
docs/governance/decision-log.md
```

Each decision should include:

* date
* decision title
* summary
* rationale
* alternatives considered
* participants
* related issues or pull requests

## Dispute Governance

Because OM World deals with intention, execution, and settlement, disputes are inevitable.

Disputes may involve:

* failed execution
* incomplete results
* fraudulent tools
* false proofs
* verifier misconduct
* unauthorized actions
* payment conflicts
* unclear mandates
* subjective outcomes

### Dispute Principles

Dispute resolution should be:

* evidence-based
* transparent where possible
* privacy-preserving where necessary
* proportional
* appealable
* resistant to collusion
* compatible with smart contract settlement

### Dispute Evidence

Disputes may use:

* intent records
* mandate records
* execution logs
* proof hashes
* transaction hashes
* verifier attestations
* tool outputs
* user submissions
* external evidence
* expert review

## Emergency Governance

Emergency governance may be needed for:

* smart contract exploits
* critical vulnerabilities
* verifier corruption
* registry attacks
* malicious tools
* large-scale privacy leaks
* governance attacks
* settlement failures

Emergency powers must be limited.

They should include:

* clear activation criteria
* limited duration
* public disclosure
* post-event review
* no permanent privilege
* no hidden fund movement
* no retroactive alteration of user intent records

Emergency actions should be documented.

## Security Review

Major governance changes should include security review.

This includes changes to:

* smart contracts
* settlement rules
* verifier incentives
* staking systems
* tool registry rules
* dispute systems
* treasury controls
* upgrade authority

Security review may include:

* internal review
* public review
* external audit
* formal verification
* adversarial testing
* bug bounty programs

OM World governance must consider attacks such as:

* token capture
* bribery
* Sybil attacks
* fake contributors
* malicious tools
* verifier collusion
* governance spam
* proposal flooding
* emergency power abuse
* treasury drain
* reputation manipulation
* social engineering
* insider privilege

## Contributor Conduct

Governance discussions should follow these norms:

* argue with evidence
* assume good faith unless proven otherwise
* disclose conflicts of interest
* separate people from ideas
* avoid personal attacks
* avoid hype
* avoid manipulation
* document claims
* respect privacy
* keep decisions public where possible

Bad behavior may include:

* harassment
* threats
* spam
* scams
* undisclosed paid influence
* manipulation of governance processes
* impersonation
* malicious misinformation
* attempts to capture user intent

## Conflict of Interest

Contributors should disclose relevant conflicts of interest.

Examples:

* financial interest in a tool
* ownership of a verifier service
* employment by a competing protocol
* investment in a model provider
* grant application under review
* relationship with a proposal beneficiary

Disclosure does not automatically disqualify participation.

But undisclosed conflicts damage governance trust.

## Relationship to Future Token

This document does not define a token.

**No token is planned at this time. No allocation of any token or financial instrument is contemplated for any party — including contributors, Genesis Builders, founders, or operators. No party has been promised, by this document or any other OM World channel, that they will receive value of any kind in any future state of the protocol.**

Any future token must be evaluated separately. If a token is ever introduced, its allocation would be decided through the then-current governance process; no retroactive commitments exist or are implied by participation in OM World today.

A token, if introduced, must serve the protocol.

It should not replace user sovereignty.
It should not override verification.
It should not create unchecked control.
It should not turn governance into speculation.

The correct order is:

1. principles
2. protocol
3. contributors
4. usage
5. verification
6. governance
7. token, if necessary

Not the reverse.

## Final Principle

Governance is not the owner of OM World.

Governance is the process by which the world remains open, verifiable, and aligned with the people who declare intentions within it.

A good governance system does not ask users to trust rulers.

It creates rules that make abuse difficult, visible, and accountable.

**One Mind, One World.**

**AI executes. Crypto verifies. Humanity owns its intentions.**
