# Contributing to OM World

## One Mind, One World.

OM World is in its Genesis phase.

This repository is the public starting point for a decentralized intent economy protocol where AI executes and crypto verifies.

We welcome contributors who want to help define, design, build, verify, explain, and govern this new world.

## Contribution Principles

### 1. Intent Sovereignty

Every contribution should respect the principle that the originator owns the intention.

Do not design systems that secretly capture, manipulate, or monetize user intent.

### 2. Verifiability

If a claim matters, it should be provable.

Prefer designs that create evidence, logs, hashes, attestations, or reproducible records.

### 3. Open Protocol, Not Closed Platform

OM World should become an open substrate.

Avoid designs that depend on a single company, single model, single interface, or single operator.

### 4. Human Agency

AI should execute, assist, and amplify.

It should not replace human consent, ownership, or responsibility.

### 5. Composability

Tools, agents, verifiers, interfaces, and economic modules should be designed as interoperable parts.

### 6. Security Before Scale

Do not optimize for growth before trust, verification, and safety are understood.

### 7. Slow on Values, Fast on Execution

Core principles should be stable.

Implementation details should iterate quickly.

## How to Start

### Step 1: Read the Core Documents

Start with:

* README.md
* MANIFESTO.md
* LITEPAPER.md
* ROADMAP.md
* BRAND.md

### Step 2: Review Open Issues

Look for labels such as:

* `genesis`
* `good first issue`
* `spec`
* `research`
* `writing`
* `design`
* `protocol`
* `agent`
* `verification`

### Step 3: Comment Before Large Changes

For major proposals, start with an issue or discussion before submitting a large pull request.

Good examples:

* “Proposal: Intent Schema v0.1”
* “Proposal: Execution Proof categories”
* “Proposal: Tool Registry metadata”
* “Threat model: malicious agent routing”
* “Design review: homepage hero section”

### Step 4: Submit a Pull Request

For document contributions:

* keep language clear
* avoid hype
* define terms precisely
* include rationale
* explain tradeoffs
* link related issues

For technical contributions:

* include tests if applicable
* document assumptions
* explain security implications
* avoid unnecessary dependencies
* keep modules small and composable

## Pull Request Guidelines

A good pull request should include:

1. What changed
2. Why it changed
3. Which issue it relates to
4. What tradeoffs exist
5. Whether the change affects terminology, architecture, security, or governance

Use clear titles.

Good examples:

* `docs: define intent lifecycle v0.1`
* `spec: add initial agent mandate format`
* `brand: add visual language PDF`
* `research: map verification methods for procurement intents`
* `governance: draft genesis contributor process`

Avoid vague titles such as:

* `update`
* `fix`
* `new stuff`
* `changes`

Please write in a style that is:

* clear

* calm
* serious
* globally understandable
* technically grounded

* hype
* pump language
* exaggerated promises
* religious claims
* vague mysticism
* “to the moon” language
* overuse of buzzwords
* unsupported technical claims

Preferred phrases:

* intent sovereignty
* verifiable execution
* open protocol
* agent mandate
* execution proof
* cryptographic settlement
* user-owned intent
* AI executes
* crypto verifies

Core principles:

* cosmic, not vague
* mystical, not religious
* technical, not cold
* decentralized, not chaotic
* protocol-grade, not marketing-driven

Avoid:

* generic AI purple gradients
* cheap crypto metal coins
* religious OM symbols
* cyberpunk neon
* neural-network spaghetti
* rocket or moon imagery

## Technical Style

When contributing technical designs:

* state assumptions
* define inputs and outputs
* separate on-chain and off-chain components
* identify trust assumptions
* identify failure modes
* include verification paths
* consider privacy
* consider abuse cases
* prefer modular architecture

## Security and Abuse Considerations

Every OM World design should consider abuse.

Potential risks include:

* malicious agents
* fake tools
* unverifiable results
* false attestations
* colluding verifiers
* hidden advertising
* intent manipulation
* privacy leakage
* unauthorized execution
* governance capture
* economic exploitation

If your contribution introduces a new mechanism, include at least a short section on risks and mitigations.
